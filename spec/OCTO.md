# OCTO — On-demand Contextual Task Orchestrator

OCTO is a multi-arm orchestration pattern built on REACH.
Where REACH is a single intentional dive into one source —
OCTO coordinates multiple arms simultaneously, surfaces a
contextual decision interface at the meaningful moment,
captures human judgment, and continues.

Named after the octopus — eight arms, each semi-autonomous,
coordinated by one intelligence, adapting its surface to context.

---

## The Octopus Metaphor

The octopus doesn't plan each arm independently.
It has an intent. The arms find the way.

Each arm is a REACH — diving into a source, surfacing with something.
The octopus holds all findings, reads the combined signal,
and presents the right surface for *this* moment.

```
Intent declared
  ↓
Arms reach simultaneously
  outlook.inbox  →  urgent items
  teams.planner  →  overdue tasks
  teams.transcript → last meeting summary
  git.commits    →  what shipped yesterday
  ↓
OCTO reads combined signal
  ↓
Contextual surface generated — not pre-built, authored for this moment
  ↓
Human decides
  ↓
Arms act on the decision
  ↓
OCTO closes — warm, with personality
```

No pre-built UI. No fixed form. The surface is always right
because it's always written for *now*.

---

## REACH vs OCTO

```
REACH  →  the language
           single arm, one source, one output
           dotnet run task.cs
           reach outlook.inbox where subject contains "[PROJECT]"

OCTO   →  the orchestrator
           multiple arms, combined signal, decision surface
           morning.octo, deploy.octo, review.octo
           the creature coordinating the arms
```

REACH is the vocabulary. OCTO is the sentence.
OCTO is built *from* REACH reaches composed together.
Neither replaces the other.

## The .octo File

An OCTO workflow lives in a `.octo` file.
Declarative. Shareable. Claude authors it.
Same philosophy as `.reach` — intent over implementation.

```
# morning.octo

name    morning-brief
arms
  - outlook.inbox     since yesterday     analyze urgent
  - teams.planner     status overdue
  - teams.transcript  latest              summarize
  - git.commits       since yesterday     analyze cadence

surface
  title   "Good morning. Here's what needs you."
  show    urgent-items
  actions
    draft-reply   →  reach outlook.draft from selected
    send-summary  →  reach outlook.draft to team summarize all
    snooze        →  reach teams.planner snooze selected
    show-all      →  expand full output

close
  tone  warm
  joke  true
```

Eight lines of intent. Claude compiles each arm to typed C#,
runs them, reads the combined signal, generates the surface,
captures input, acts, closes with personality.

## The Contextual Surface

The surface is not a pre-built form.
It is generated by Claude for *this specific moment* —
the right title, the right items, the right actions,
based on what the arms actually found.

**Structure:**

```
┌──────────────────────────────────────────┐
│  [title — generated from findings]       │
│                                          │
│  [item 1 — most urgent]                  │
│  [item 2]                                │
│  [item 3]                                │
│                                          │
│  [action 1]  [action 2]  [action 3]      │
│                                          │
│  [secondary action]  [dismiss]           │
└──────────────────────────────────────────┘
```

**Generated, not fixed:**
- Title reflects what was actually found
- Items are ranked by signal strength
- Actions are relevant to the items shown
- Secondary actions offer escape routes

## The Morning Brief — Reference Demo

The canonical OCTO demonstration. Five arms. One surface. One decision. One joke.

**The arc:**

```
08:45 — morning.octo runs (Task Scheduler or manual)

Arms reach simultaneously:
  outlook.inbox    → 3 flagged items, 1 urgent thread
  teams.planner    → 2 overdue, 1 due today
  teams.transcript → last standup: deployment mentioned
  git.commits      → 2 commits yesterday, on track

OCTO reads combined signal:
  Urgent: Sarah emailed twice, no reply
  Overdue: project planner item 3 days past due
  Context: deployment discussed in standup

Surface appears:
┌──────────────────────────────────────────────┐
│  Good morning. Here's what needs you.        │
│                                              │
│  🔴 Sarah emailed twice — awaiting reply     │
│  🟡 Project planner item — 3 days overdue     │
│  📋 Deployment flagged in yesterday's standup│
│                                              │
│  [Draft reply to Sarah]                      │

REACH reads the full thread
```

The close is not optional. Personality is what separates
a tool from something you want to use again tomorrow.

## OCTO Arms — The Building Blocks

An arm is a named REACH sequence.
Arms run in parallel where possible, in sequence where dependent.

**Arm types:**

```
read arm    →  reaches into a source, returns findings
               no side effects, runs freely
               outlook.inbox, git.commits, teams.transcript

act arm     →  changes state after human approval
               always gated by surface confirmation
               outlook.draft → send, teams.planner → update

signal arm  →  writes a signal for another arm or process
               process.signal "urgent", reach file.watch
```

**Arm declaration:**

```
arms
  - source          qualifier         intent
  - outlook.inbox   since yesterday   analyze urgent
  - teams.planner   status overdue
  - git.commits     since yesterday
```

Claude compiles each arm independently.
Results held in OCTO context until surface is generated.

## The Decision Surface — Interaction Patterns

Four surface types — same as REACH prompt layer,
but OCTO generates the content dynamically from arm findings:

### `surface.brief`
Compact. Three items max. Fast acknowledgement.
Morning brief, deployment check, quick status.

```
surface
  type    brief
  title   auto
  items   3
  actions approve | snooze | expand
```

### `surface.review`
Detailed. Full findings visible. Multiple action paths.
Sprint review, incident response, timesheet approval.

```
surface
  type    review
  title   auto
  show    all-findings
  actions
    approve   →  act arm
    edit      →  return to draft
    delegate  →  reach outlook.draft forward
    archive   →  reach file.write to archive
```

### `surface.draft`
Email or message preview before sending.
Full content visible. Edit inline. Send or save.

```
surface
  type    draft
  show    email-preview
  actions send | edit | save-draft | discard
```

### `surface.acknowledge`
Simplest gate. Information delivered. One action to continue.
Notification, alert, status update.

```
surface
  type    acknowledge
  show    summary
  actions ok | snooze | escalate
```

## The Close

Every OCTO session ends with a close.
Not a system message. Not a log entry.
A human moment — warm, brief, occasionally funny.

```
close
  tone    warm          # warm | neutral | minimal
  joke    true          # true | false
  summary true          # recap what was done
```

**Examples:**

```
# After morning brief with reply sent:
"Sarah will appreciate that.
 One less thing before your first meeting."

# After sprint review — all red:
"Ten weeks at redline. The data doesn't lie.
 Take the valley week. You've earned it."

# After timesheet posted:
"Logged. [TIMESHEET] has been fed.
 It was hungry."

# After deployment verified — all green:
"Green across the board.
 Rare. Enjoy it."
```

Claude authors the close based on what actually happened.
The tone instruction shapes it. The joke flag adds levity.
Neither is mandatory — `tone minimal` produces a clean exit.

```

  tone    neutral
  joke    false
```

## OCTO in the Semantic Intent Ecosystem

```
CAL     →  decides          cascade analysis, scoring
RECALL  →  publishes        structured documents
EMBER   →  remembers        typed artifacts, agent handoffs
REACH   →  reaches          single arm, one source
OCTO    →  orchestrates     multiple arms, decision surface
Mere    →  contains         the file is the app
```

OCTO sits at the orchestration layer — above REACH,
below a full agent framework. Deliberately between them.

Not LangChain. Not AutoGen. Not a framework you install.
A pattern you describe in a `.octo` file and Claude runs.

## Design Principles

**1. Arms are independent**
Each arm reaches without knowing what the others found.
OCTO holds the combined signal. Arms don't talk to each other.

**2. The surface is always generated**
Never pre-built. Never a fixed form. Claude reads the findings
and authors the right surface for this moment.

**3. Read arms run freely. Act arms need approval.**
Same principle as REACH. The surface is the gate.
Nothing changes state without human confirmation.

**4. The close has personality**
Every session ends with a human moment.
A tool you use once. Something with personality you use daily.

**5. .octo files are the sharing unit**
Share a `.octo` file. Someone runs it against their data.
The arms adapt — same pattern, different context.

**6. Show capability, not complexity**
The morning brief demo is the spec.
Everything else follows from whether that lands.

---

## What OCTO Is Not

- Not an agent framework — no role definitions, no agent topology
- Not a workflow engine — no node graphs, no visual builders
- Not a chatbot — not conversational, task-oriented
- Not a dashboard — surfaces appear and disappear, nothing persists on screen
- Not infrastructure — no server, no always-on process, no install

OCTO is a pattern. A `.octo` file is the artifact.
Claude is the compiler. Your machine is the stage.
